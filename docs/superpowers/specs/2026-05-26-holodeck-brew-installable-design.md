# Holodeck Homebrew Installability

**Status:** Draft (spec)
**Author:** Eduardo Arango (eduardoa@nvidia.com)
**Date:** 2026-05-26
**Tracks:** Holodeck v0.3.x distribution — macOS/Linux install UX

## Problem

Holodeck has no managed install path. The README's only install instructions are:

```bash
make build
sudo mv ./bin/holodeck /usr/local/bin/holodeck
```

This requires a Go toolchain, a `sudo` mv, and gives the user no upgrade story.
Two compounding gaps:

1. **No release pipeline.** Tags `v0.3.2` through `v0.3.4` exist but have **zero
   attached release assets** — the `make release` target cross-builds the
   binaries locally but never publishes them. There is no `release` workflow in
   `.github/workflows/`.
2. **No Homebrew formula.** macOS users (the largest non-CI install base) have
   no `brew install` path. Compared to peer tools (kind, kubectl, terraform,
   gh) this is the single biggest install-UX gap.

## Goal

Ship holodeck via Homebrew with a one-line install for macOS and Linux users,
backed by a fully automated release pipeline. After this work, a clean machine
installs holodeck with:

```bash
brew tap nvidia/holodeck https://github.com/NVIDIA/holodeck
brew install nvidia/holodeck/holodeck
holodeck --help
```

Install must complete in seconds (no Go toolchain build), every tagged release
must auto-update the formula, and `brew upgrade` must work.

## Non-Goals

- Submitting to `homebrew-core` upstream (high notability bar, slow review,
  separable follow-up).
- Pre-built binary bottles served by NVIDIA-controlled infrastructure (the
  formula points at GitHub Release archives — the standard tap pattern).
- Distributing the `cmd/action` binary via Homebrew (it runs inside GitHub
  Actions runners, not on user machines — kept as a release asset only).
- Signed binaries (cosign/sigstore). Flagged as a likely follow-up.
- Cleaning up the stale hardcoded version string at `cmd/cli/main.go`
  (`c.Version = "0.2.18"`) as a separate concern — though the GoReleaser
  ldflags side-effect this work introduces will fix it.

## Design Decisions

Three orienting choices, locked in during brainstorming:

| Decision | Choice | Rationale |
|---|---|---|
| Tap location | `Formula/holodeck.rb` in `NVIDIA/holodeck` main branch | Avoids creating a separate `homebrew-holodeck` repo. Trade-off: `brew tap` clones the whole Go source incl. `vendor/` — user accepted this for the one-repo simplicity. |
| Binary source | Pre-built binaries from GitHub Release archives | Smoothest UX (~5s install, no Go toolchain). Requires release pipeline, which we are building anyway. |
| Release automation | GoReleaser via GitHub Action | Single tool handles cross-build, archive, checksum, GitHub Release publish, and Homebrew formula generation. Replaces the existing `make release` target. |

## Architecture

Five deliverables, all in `NVIDIA/holodeck` main branch:

### 1. `.goreleaser.yaml`

Root-of-repo GoReleaser config. Builds two binaries:

- `holodeck` from `cmd/cli` for `{linux,darwin} × {amd64,arm64}` — goes into
  the Homebrew formula.
- `holodeck-action` from `cmd/action` for `linux × {amd64,arm64}` — release
  asset only, not exposed to brew.

Both built with `CGO_ENABLED=0` (static binaries). The CLI ldflags inject
`-X main.version={{.Version}}`, `-X main.commit={{.Commit}}`,
`-X main.date={{.Date}}`. This side-effects a fix for the stale hardcoded
version at `cmd/cli/main.go` (which currently reads `c.Version = "0.2.18"`
regardless of the actual tag) — the `main` package must expose a `version`
variable for the ldflag to bind.

Archives: tar.gz, naming `holodeck_{{.Version}}_{{.Os}}_{{.Arch}}` for the
CLI, `holodeck-action_{{.Version}}_{{.Os}}_{{.Arch}}` for the action.
CLI archive bundles `LICENSE` + `README.md`. Action archive bundles `LICENSE`
only.

Checksums: single `checksums.txt` (SHA256). Source archive enabled so
build-from-source remains a published, hash-verified path.

Changelog: GoReleaser's `github` mode, excluding `^docs:`, `^chore:`, `^test:`
prefixes.

The `brews:` block writes `Formula/holodeck.rb` back to the same repo's main
branch via the workflow's `GITHUB_TOKEN`. Multi-arch formula structure
auto-generated by GoReleaser (`on_macos`/`on_linux` × `on_arm`/`on_intel`).

### 2. `.github/workflows/release.yaml`

Triggers on `push: tags: ['v*']`. Permissions `contents: write` (publish
release + commit formula). Steps:

1. `actions/checkout@v4` with `fetch-depth: 0` (GoReleaser needs full history
   for changelog generation).
2. `actions/setup-go@v5` with `go-version-file: go.mod` + module cache.
3. `goreleaser/goreleaser-action@v6` with `args: release --clean`,
   `GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}`.

Pinned-action major versions verified against the existing workflows in
`.github/workflows/` during plan execution.

### 3. `Formula/holodeck.rb`

Auto-generated and auto-maintained by GoReleaser on each tag. Shape:

```ruby
class Holodeck < Formula
  desc "Tool for creating and managing GPU-ready cloud test environments"
  homepage "https://github.com/NVIDIA/holodeck"
  version "0.3.5"
  license "Apache-2.0"

  on_macos do
    on_arm do
      url ".../holodeck_0.3.5_darwin_arm64.tar.gz"
      sha256 "..."
    end
    on_intel do
      url ".../holodeck_0.3.5_darwin_amd64.tar.gz"
      sha256 "..."
    end
  end
  on_linux do
    on_arm do
      url ".../holodeck_0.3.5_linux_arm64.tar.gz"
      sha256 "..."
    end
    on_intel do
      url ".../holodeck_0.3.5_linux_amd64.tar.gz"
      sha256 "..."
    end
  end

  def install
    bin.install "holodeck"
  end

  test do
    system "#{bin}/holodeck", "--help"
  end
end
```

**Bootstrap:** the first commit lands the GoReleaser config + workflow, then
tags `v0.3.5`. GoReleaser produces the archives and the first
`Formula/holodeck.rb` in that release cycle. v0.3.4 is not retroactively
backfilled — the gap is short and adding phantom assets to an existing
release is brittle.

### 4. README install section

New "Install via Homebrew" subsection added to the README's install area:

```bash
brew tap nvidia/holodeck https://github.com/NVIDIA/holodeck
brew install nvidia/holodeck/holodeck
holodeck --help
```

The existing `make build` flow is retained as the contributor / source-install
path. The custom-URL `brew tap` form is required because the formula lives in
`NVIDIA/holodeck` (which doesn't match the `homebrew-*` repo-name
convention) instead of a dedicated `NVIDIA/homebrew-holodeck`.

### 5. `Makefile` `snapshot` target

One-line addition:

```makefile
snapshot:
	goreleaser release --snapshot --clean --skip=publish
```

Lets maintainers run a dry-run release locally before tagging, producing
`dist/` artifacts and a draft formula without touching GitHub.

The existing `release:` target is **removed** (replaced by GoReleaser).
Its multi-OS/arch loop is now `.goreleaser.yaml`'s `builds:` block.

## Auth Model

GoReleaser pushes the formula commit to the same repo it was triggered from.
The auto-issued `GITHUB_TOKEN` with `contents: write` permission both
publishes the release and commits `Formula/holodeck.rb` back to main.
**No PAT (Personal Access Token) is required.** This is the key benefit of
the same-repo tap choice.

**Branch-protection risk:** if `main` has required-PR or required-status-check
protection, the direct commit will fail. Two options at plan time:

- Confirm branch protection allows admins / bot writes, or
- Switch `brews.repository` to PR mode (`pull_request: enabled: true`), which
  requires a PAT because the default `GITHUB_TOKEN` cannot approve/merge its
  own PR.

The plan's first task is to inspect the repo's branch-protection settings and
pick the path. If a PAT is needed, scope it to `contents: write` on
`NVIDIA/holodeck` only, store as a repo secret named
`HOMEBREW_TAP_GITHUB_TOKEN`, and reference it from the workflow.

## Testing Strategy

Three layers:

1. **Local snapshot test** — `make snapshot` runs `goreleaser release
   --snapshot --clean --skip=publish` and produces `dist/` with archives and a
   draft `Formula/holodeck.rb`. CI runs this as a job on PRs that touch
   `.goreleaser.yaml`, `Formula/**`, or the release workflow itself.

2. **Formula syntax validation** — a `homebrew-validate` job runs
   `brew audit --strict --online Formula/holodeck.rb` and
   `brew style Formula/holodeck.rb` on every push to main when `Formula/**`
   changes. Catches Ruby/formula syntax errors before users hit them.

3. **Post-release smoke test** — manual checklist documented in
   `docs/release.md`: on a clean macOS arm64, macOS amd64, and Linux machine
   (or GH-hosted runners), run `brew tap nvidia/holodeck
   https://github.com/NVIDIA/holodeck && brew install
   nvidia/holodeck/holodeck && holodeck --help`. Automatable later via a
   matrix `post-release.yaml`; flagged as a follow-up.

**Explicitly not tested by this work:**

- Holodeck command functional coverage — the existing test suite owns that.
- `brew upgrade` cross-version paths — Homebrew handles this; we trust it.
- Bottle distribution — we do not produce bottles, only archives.

## Failure Modes

| Mode | Surface | Recovery |
|---|---|---|
| Tag pushed without release notes / unclean working tree | GoReleaser fails fast in CI | Delete the tag, fix, re-tag |
| Formula commit fails (branch protection) | Release succeeds, formula stays at previous version | Surfaced by `homebrew-validate` job on next push; hand-merge the formula update; switch to PR mode |
| Archive URLs change shape (GoReleaser major version upgrade) | Formula points at 404 URLs | Pin `goreleaser-action` to `~> v2`; review on major bumps |
| Future tag publishes a broken binary | Brew install installs the broken binary | `holodeck --help` smoke test in the post-release checklist catches before announcement; revoke release if needed |
| `~/.config/goreleaser`-style local state contamination | Snapshot succeeds locally but fails in CI | `--clean` flag in both local and CI invocations forces a clean `dist/` |

## Rollout

1. Land `.goreleaser.yaml`, `release.yaml` workflow, `Makefile` snapshot
   target, README install section, and a bootstrap `Formula/holodeck.rb`
   (hand-written, pointing at v0.3.5 archives the release will produce) in
   one PR.
2. Verify `make snapshot` locally — inspect `dist/Formula/holodeck.rb`
   matches the hand-written bootstrap shape.
3. Squash-merge to main.
4. Tag `v0.3.5`, push tag, watch the `release` workflow.
5. Verify release page has 6 archives (4 CLI: linux/darwin × amd64/arm64;
   2 action: linux × amd64/arm64) + `checksums.txt` + source tarball,
   and that GoReleaser committed `Formula/holodeck.rb` to main matching the
   release.
6. Run the post-release smoke test (clean macOS arm64 minimum).
7. Announce in the README and (optionally) a discussion / release-notes
   highlight.

## Open Questions (resolve at plan time)

1. **Branch protection on `main`** — does it allow the workflow's
   `GITHUB_TOKEN` direct-commit, or do we need PR mode + PAT?
2. **`cmd/cli/main.go` version variable** — does a `version` variable exist
   in `package main` for `-ldflags -X main.version=...` to bind? If not, a
   tiny `var version = "dev"` declaration is added as part of this work
   (in-scope because the ldflags depend on it).
3. **Action version pins** — match the style of the existing workflows
   (`@v4` vs `@sha-pin`); verify before writing the workflow file.

## Out of Scope (explicit follow-ups)

- Homebrew-core submission.
- Signed binaries (cosign / sigstore attestation).
- Automated post-release smoke test matrix workflow.
- Migrating to a dedicated `NVIDIA/homebrew-holodeck` tap repo if tap-clone
  size becomes a complaint.
- SBOM publishing alongside release archives.
